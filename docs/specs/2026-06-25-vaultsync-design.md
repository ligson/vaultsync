# VaultSync 私有文件同步系统设计稿

## 1. 背景

VaultSync 是一套面向 1-3 人小规模使用的私有文件同步系统，部署在 NAS 上，支持桌面端和移动端客户端。核心目标是：

- 客户端可选择同步文件夹
- 文件在服务器端仅以密文形式保存
- 不同用户完全隔离
- 上传成功后可配置本地清理策略

本设计基于需求文档 V1.1，优先满足单机 NAS 部署与端到端加密。

## 2. 设计原则

- 服务器不持有明文文件内容
- 服务器不持有明文密钥
- 文件名、路径、目录结构也应加密或混淆
- 默认私有隔离，不设计跨用户共享为首要能力
- 先做可用闭环，再补高级同步体验

## 3. 推荐技术栈

- 后端：Go
- 元数据：SQLite + WAL
- 文件存储：NAS 本地文件系统或挂载卷
- 客户端：后续可选 Flutter 或 Tauri/Electron
- 加密：客户端侧 AEAD，例如 AES-256-GCM 或等价方案

## 4. 总体架构

### 4.1 客户端

客户端负责：

- 用户登录
- 绑定设备
- 选择同步文件夹
- 监控文件变更
- 上传前加密
- 下载后解密
- 按策略处理本地文件

### 4.2 后端服务

后端负责：

- 认证与设备管理
- 目录订阅与同步状态管理
- 密文对象接收与下载
- 版本记录
- 冲突标记
- 管理员级别的系统监控

### 4.3 存储层

分两类：

- SQLite：用户、设备、同步目录、版本、游标、任务记录
- 文件系统：密文文件、加密元数据、版本对象

## 5. 核心数据概念

- User：用户账号
- Device：用户设备
- Sync Root：用户选择的同步目录
- Object：加密后的文件对象
- Version：对象版本
- Change Cursor：同步游标
- Policy：上传后本地处理策略

## 6. 加密方案

### 6.1 目标

- 服务器无法解密用户数据
- 服务器无法从文件名和路径轻易恢复业务结构
- 用户只要保住密码或设备密钥，就能恢复访问

### 6.2 推荐模型

- 用户输入密码或使用设备派生密钥
- 客户端派生主密钥
- 主密钥再派生文件夹密钥和文件密钥
- 文件内容、元数据分别加密

### 6.3 说明

第一版优先保证安全边界清晰，不追求复杂的密钥共享和多方协作。

## 7. 同步流程

### 7.1 上传

1. 客户端扫描变化
2. 生成加密内容与加密元数据
3. 上传到后端
4. 后端写入密文对象和 SQLite 索引
5. 客户端确认成功后执行本地策略

### 7.2 下载

1. 客户端拉取远端变更
2. 下载密文对象
3. 客户端解密并恢复本地文件
4. 更新本地同步状态

### 7.3 冲突

第一版采用保守策略：

- 发现冲突时保留双方版本
- 由客户端生成冲突副本或提示用户处理
- 不在服务端做自动合并

## 8. 本地处理策略

每个同步目录可配置以下一种策略：

- 保留本地文件
- 成功上传后删除本地文件
- 成功上传后移动到归档目录

移动端优先用于相机胶卷自动备份与清理。

## 9. 后端模块

- Auth：登录、令牌、2FA 扩展位
- Device：设备注册与解绑
- Folder：同步目录订阅
- Sync：游标、变更、冲突
- File Object：上传、下载、版本
- Admin：系统监控、日志、用户管理

## 10. 数据库方向

建议的 SQLite 核心表：

- users
- devices
- sync_roots
- file_objects
- file_versions
- sync_cursors
- upload_sessions
- audit_logs

## 11. 部署方式

- Docker / Docker Compose
- 单机 NAS 部署
- 卷挂载到持久化存储
- HTTPS 通过反向代理或内置服务支持

## 12. 第一阶段范围

必须做：

- 登录与设备绑定
- 同步目录选择
- 密文上传与下载
- 版本记录
- 本地清理策略
- 单机部署

暂不做：

- 多实例与集群
- 跨用户共享
- 完整文件浏览管理界面
- 桌面虚拟文件系统
- 复杂后台调度

## 13. 验收标准

- 用户可以选择同步目录
- 服务器仅保存密文
- 不同用户互相不可见
- 上传后本地处理策略可生效
- 单机 NAS 可稳定运行
